# Worked Examples

> CLI-first / CLI 优先：示例里的推荐命令优先展示 `scripts/inkos_cli.py`，底层脚本调用仅作为补充。

这份文件不是理论说明，而是给 agent 和人类作者看的“落地示范”。

目标：
- 让使用者理解这个 skill 到底怎么从 0 跑起来
- 让 workflow 不再只是抽象步骤
- 给 chapter planning / audit / state update 提供可模仿范式

---

## Example 1: 从 0 启动一个悬疑向网文项目

### 用户意图

> 帮我搭一个能长期写的悬疑长篇项目，主角是一个冷静但多疑的古风男主，故事核心围绕一块被调包的玉佩展开。

### 推荐动作

1. 初始化项目骨架
2. 先填最小 truth files
3. 明确第一阶段主线冲突
4. 不急着写正文，先让状态文件成形

### 初始化命令

```bash
bash scripts/init_novel_project.sh /path/to/project "玉佩疑云"
```

### 初始化后至少先补这些文件

#### `story_bible.md`
最少补：
- 世界背景
- 核心悬念
- 玉佩相关规则
- 关键势力
- 关键地点

#### `book_rules.md`
最少补：
- 主角性格锁：冷静、多疑、不轻信
- 不允许轻易摊牌
- 悬疑线索必须分层释放
- 不能靠旁白直接把真相端出来

#### `outline.md`
最少补：
- 第一阶段：玉佩异样 → 怀疑徐安 → 顺藤摸到旧案
- 中期：怀疑对象转移，真相变复杂
- 后期：玉佩只是入口，牵出更大的局

#### `current_state.md`
最少补：
- 当前时间点
- 主角已知事实
- 主角怀疑对象
- 徐安当前立场不明
- 玉佩在主角手中

## Example 2: 写第 1 章之前怎么准备

### 用户意图

> 现在帮我开始写第一章，重点是让主角第一次察觉玉佩不对劲。

### 推荐工作流

1. 读：`story_bible.md` / `book_rules.md` / `outline.md` / `current_state.md`
2. 写出 chapter function
3. 明确这章不能做什么
4. 再开始起草

### 好的 chapter function 示例

- 让主角第一次明确感到“玉佩不对”
- 建立徐安在场但态度暧昧
- 留下“谁先碰过玉佩”的悬念
- 不能在第一章直接确认幕后黑手

### 起草时的约束

- 重点写“察觉”而不是“揭秘”
- 重点写具体手感、旧物痕迹、人物试探
- 不要让主角突然得到完整真相
- 不要为了制造钩子而让所有人一起震惊

## Example 3: 写完第 1 章后如何审计

### 用户意图

> 这一章写完了，帮我看有没有问题。

### 审计命令

```bash
python3 scripts/audit_chapter.py \
  --project /path/to/project \
  --chapter-file /path/to/project/chapters/ch01.md
```

### 应重点看什么

- 主角有没有 OOC
- 是否泄露了超出 POV 的信息
- 是否提前说破悬念
- 是否有“报告感”太强的句子
- 节奏是不是一上来就打满

### 审计后好的处理方式

优先：
- spot-fix 某几句
- 降低明说程度
- 把解释改成动作 / 细节 / 对话

避免：
- 因为一处气口不对就重写整章

## Example 4: 接受章节后如何更新状态

### 用户意图

> 第一章定稿了，记住这一章发生了什么。

### 更新命令

```bash
python3 scripts/update_story_state.py \
  --project /path/to/project \
  --chapter 1 \
  --title "第一章" \
  --summary "林烬察觉玉佩手感异常，开始怀疑有人提前动过手脚。" \
  --state-change "林烬确认玉佩并非自己记忆中的原物。" \
  --hook-open "是谁在林烬之前碰过玉佩" \
  --relationship "林烬 -> 徐安：试探增加，信任下降" \
  --emotion "林烬：警觉上升"
```

### 更新后还应人工检查

- `current_state.md` 是否写清“主角知道什么、不知道什么”
- `pending_hooks.md` 是否把新悬念登记进去
- `character_matrix.md` 是否体现关系张力变化

## Example 5: 从第 1 章推进到第 3 章

### 目标节奏

#### 第 1 章
- 主角察觉玉佩不对
- 徐安在场但态度模糊
- 埋下“有人动过玉佩”的疑问

#### 第 2 章
- 主角试探徐安或旁敲侧击
- 新证据不完全支持初始怀疑
- 悬疑从“物件异常”转向“人际不信任”

#### 第 3 章
- 怀疑线出现第一次偏转
- 读者意识到徐安未必是最直接的嫌疑人
- 但主角的信任裂缝已经形成

### 好的三章推进特点

- 每章都推进至少一个 hook
- 每章都改变一点状态
- 每章都留下下一章为什么还要继续看的理由
- 真相不是直线揭露，而是不断校正怀疑对象

### 坏的三章推进特点

- 第 1 章就把真相说穿
- 第 2 章纯解释背景，没有状态变化
- 第 3 章靠新人物硬扔答案
- 角色怀疑 / 信任变化没有记录进 truth files

## Example 6: 什么时候该重写，什么时候只该 spot-fix

### 只该 spot-fix 的情况

- 某几句像 AI 报告
- 某段解释太直白
- 段落节奏太平均
- 个别群体反应过于 cliché

### 该重写场景的情况

- 本章核心冲突立错了
- 主角性格锁被根本破坏
- 章节结果与 `outline.md` / `current_state.md` 正面冲突
- 信息边界严重泄漏，导致悬疑结构塌掉

### 判断原则

> 小问题修句，大问题修场景；不是所有不顺眼都值得重写。
